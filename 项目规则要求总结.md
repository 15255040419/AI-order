# 下单助手业务规则要求总结

本文档根据当前项目代码、配置文件和现有说明整理，记录下单助手目前实际执行的业务规则。

## 1. 系统总体原则

下单助手不是完全依赖 AI 的系统，而是采用“AI 解析 + 本地硬规则兜底”的方式处理订单。

核心原则：

- AI 主要负责理解订单原文，解析收货人、电话、地址、省份、货品、数量、价格等信息。
- 本地规则优先处理付款、收款账户、业务员、客户账号、货品匹配、快递判定和备注合成。
- AI 解析结果必须经过库存表、客户档案、快递表格和配置规则二次校验。
- 遇到关键业务字段时，以本地硬规则为准，必要时覆盖 AI 结果。

## 2. 订单处理流程

当前订单处理流程如下：

1. 用户输入订单原文。
2. 后端先用本地规则预解析手机号、客户、规格编号货品、付款方式和收款账户。
3. 将订单原文和已知信息交给 AI 解析。
4. AI 返回结构化订单信息。
5. 系统用库存表匹配标准货品名称，并补全规格编号、货品编号、条码、规格、重量等信息。
6. 系统强制应用本地规则，生成业务员、客户账号、客服备注、付款方式、收款账户等字段。
7. 根据省份、货品类型和重量判定快递。
8. 前端展示订单卡片，用户可人工修改。
9. 导出 Excel，生成订单表和货品表。

## 3. AI 解析规则

AI 输出固定 JSON 格式，字段包括：

- 收货人
- 手机号
- 收货地址
- 省份
- 货品列表
- 付款状态
- 收款账户
- 客户账号
- 业务员代码
- 邮费
- 总金额
- 备注

AI 解析要求：

- 货品名必须尽量完整保留型号和规格，例如 `JY-335C`。
- 优先对齐系统提供的候选货品。
- 如果本地预解析已经提供字段，AI 应保持一致。
- AI 只返回 JSON，不输出解释文字。

## 4. 业务员规则

业务员优先从订单原文括号中识别代码，例如 `(L)`、`（D）`。

当前业务员代码映射：

| 代码 | 业务员 |
| --- | --- |
| TX | 仝心科技(admin) |
| L | 陆香(12) |
| D | 王德龙(21) |
| C | 王德成(31) |
| W | 汪朋松(30) |
| T | 胡正婷(16) |
| H | 韩伟(13) |
| F | 方同辉(14) |

规则：

- 如果原文括号内存在有效业务员代码，优先使用该业务员。
- 如果没有有效代码，使用 AI 或客户档案中的业务员。
- 如果仍无法识别，默认业务员为 `仝心科技(admin)`。

## 5. 客户账号规则

客户账号优先从订单原文中识别。

系统会识别以下关键词后面的内容：

- `系统录:`
- `系统录入:`
- `系统录制:`
- `系统记录:`

规则：

- 原文中出现上述关键词时，关键词后紧跟的内容作为客户账号。
- 该规则优先级高于 AI 解析结果。
- 如果原文没有系统录账号，则使用 AI 解析出的客户账号。
- 客户档案会按客户账号、客户名称、手机号建立索引，用于辅助识别老客户。

## 6. 客户档案规则

客户档案来源为 `客户档案.xlsx`。

主要字段：

- 客户编号
- 客户名称
- 联系人
- 联系电话
- 联系地址
- 业务员
- 客户账号
- 来源渠道

当前规则：

- 系统会根据订单原文中的手机号、客户名称或客户账号匹配客户档案。
- 如果匹配到客户档案，并且当前订单没有业务员，则可使用客户档案中的业务员。
- 客户档案目前不覆盖客服备注。

## 7. 付款方式规则

付款方式由本地规则判断，不依赖 AI。

规则：

- 原文包含 `已付`、`转账`、`已转`，结算方式为 `银行收款`。
- 原文包含 `未付`、`欠款`，结算方式为 `欠款计应收`。
- 如果没有明确付款信息，默认结算方式为 `欠款计应收`。
- 如果识别到了具体收款账户，结算方式强制为 `银行收款`。
- 前端手动将结算方式改为 `欠款计应收` 时，会清空收款账户。
- 前端选择收款账户时，会自动把结算方式改为 `银行收款`。
- 前端清空收款账户时，会自动把结算方式改为 `欠款计应收`。

## 8. 收款账户规则

当前收款账户映射：

| 原文关键词 | 收款账户 |
| --- | --- |
| 已付仝心农商 | 仝心农商（公账） |
| 已付农商成 | 农商银行（成） |
| 已付农商龙 | 农商（龙） |
| 已付农商 | 农商银行（成） |
| 已付财务微信 | 财务微信 |
| 已付安徽仝心 | 仝心农商（公账） |
| 已付仝心 | 仝心农商（公账） |

规则：

- 系统按关键词长度倒序匹配，避免短关键词误伤长关键词。
- 只要原文命中收款账户关键词，就认为订单为 `银行收款`。
- 如果判断为 `银行收款`，但没有匹配到具体账户，则默认取配置中的第一个收款账户。

## 9. 金额和价格规则

当前规则：

- 如果订单原文中出现 `xxx元`，系统优先取该金额作为订单总金额。
- 如果原文没有金额，系统按货品 `单价 * 数量` 求和。
- 货品单价主要来自 AI 解析结果。
- 库存表目前用于匹配货品和补全信息，不用于自动取销售价格。
- 前端手动修改货品价格、数量或货品名称后，会重新计算订单合计。
- 应收邮资默认为 `0`。

注意：

- 当前系统没有从库存表中的成本价或当前成本价自动填销售单价。
- 如果希望按客户、货品或价格表自动定价，需要后续单独补规则。

## 10. 货品匹配规则

货品匹配优先级：

1. 配置文件中的货品特殊映射。
2. 后续学习到的货品映射规则。
3. 规格编号精确匹配。
4. 货品名称精确匹配。
5. 包含匹配。
6. 模糊匹配。

匹配细节：

- 系统会将查询货品转为大写并去除首尾空格。
- 如果原文货品名命中特殊映射，直接使用映射后的标准货品名。
- 如果命中规格编号，直接找到对应库存货品。
- 如果原文包含库存货品名，或者库存货品名包含原文关键词，会进行包含匹配。
- 包含匹配时，优先选择名称最短的匹配项，通常认为它更接近基础型号。
- 最后才使用模糊匹配。

## 12. 货品信息回填规则

货品匹配成功后，系统从库存数据中补全：

- 标准货品名称
- 规格编号
- 货品编号
- 条码
- 规格
- 重量

库存数据来源：

- `总库存.xlsx`
- `总库存-组合装明细.xlsx`

前端手动切换货品名称时，也会尝试从本地缓存中同步补全条码、规格编号和规格。

## 13. 客户购买习惯规则

项目中存在 `customer_history.json`，用于记录客户历史购买习惯。

记录内容可能包括：

- 标准货品名称
- 规格编号
- 货品编号
- 条码
- 规格

当前状态：

- 文件中已有部分客户购买记录。
- 但当前主处理流程没有完整接入该历史购买习惯文件。
- 因此它目前更像是预留能力，还不是稳定生效的强规则。

## 14. 学习规则

前端支持在人工修正后触发学习。

当前支持方向：

- 货品学习：原始货品名 -> 标准货品名。
- 客户学习：原始客户标识 -> 标准客户账号。
- 快递学习：前端有触发逻辑，但后端当前没有完整接入快递学习规则。

规则：

- 后端会将学习结果保存到 `learn_rules.json`。
- 启动时如果存在 `learn_rules.json`，会将其中的货品和客户学习规则合并到配置中。
- 当前项目目录中暂未发现已有 `learn_rules.json`。

## 15. 地址识别规则

地址由 AI 解析，本地做轻量清洗。

规则：

- 地址首尾的空格、逗号、句号、分号等符号会被清理。
- 省份优先使用 AI 提取结果。
- 如果 AI 省份无法匹配，则从地址正文中查找省份关键词。
- 支持识别常见中国省份、直辖市和自治区。

## 16. 快递判定总规则

快递判定优先级：

1. 原文手动指定快递。
2. 省份快递表规则。
3. 货品品类规则。
4. 重量分段规则。
5. 默认快递。

当前快递选项：

- 中通（渠道）
- 圆通（渠道）
- 德邦特惠
- 极兔渠道（新）
- 虚拟单号
- 顺丰到付（渠道）
- 顺丰现付（渠道）
- 工厂代发
- 工厂直发

如果省份无法匹配快递表，默认使用 `中通（渠道）`。

## 17. 手动指定快递规则

系统会在订单原文中识别用户是否指定快递。

支持关键词：

- `发`
- `走`
- `指定`
- `要`
- `送`
- `快递`
- `备注`

支持快递映射：

| 原文关键词 | 物流公司 |
| --- | --- |
| 顺丰 | 顺丰现付（渠道） |
| 圆通 | 圆通（渠道） |
| 中通 | 中通（渠道） |
| 极兔 | 极兔渠道（新） |
| 德邦 | 德邦特惠 |
| 工厂 | 工厂直发 |

防误判规则：

- 快递判定前，系统会从原文中扣除收货地址和收货人。
- 这样可以避免地址中的“圆通路”“中通街”等字样被误判为指定快递。

## 18. 品类判定规则

当前主要区分三类：

- 一体称
- 收银机
- 普通货品

### 一体称

判定规则：

- 货品名以 `X` 后跟数字开头。
- 或货品名包含 `一体称`。

快递规则：

- 一体称优先走快递表中的一体称列。
- 一体称通常发 `德邦特惠`。

### 收银机

判定规则：

- 货品名以 `K` 后跟数字开头。
- 或货品名包含收银机相关词。

排除规则：

以下词命中时，不按收银机大件处理：

- 适配器
- 色带
- 纸
- 摄像头

快递规则：

- 收银机按快递表中的收银机列，不按普通重量分段。

### 普通货品

普通货品按重量分段：

- `<= 2kg`
- `2kg - 5kg`
- `> 5kg`

然后根据省份查快递表对应列。

## 19. 快递表格规则

快递规则主要来源为 `快递表格.xls`。

表格中主要列包括：

- 省份
- 打印机 1.01-2KG
- 打印机 2kg-5kg
- 打印机 5kg以上
- 收银机
- 一体称
- 其他可发快递

系统根据省份和品类读取对应列。

注意：

- `配置规则.json` 中也存在一份快递规则配置。
- 但当前实际运行时主要使用 `快递表格.xls` 中的规则。

## 20. 客服备注规则

客服备注由系统强制合成，不直接使用 AI 备注。

格式：

```text
货品原始摘要 + 原文备注内容
```

货品摘要格式：

```text
原始货品名*数量
```

多个货品摘要之间使用 `+` 连接：

```text
原始货品A*数量+原始货品B*数量
```

示例：

```text
JY-335C黑色*2 发黑色
```

规则：

- 系统使用 AI 提取时保留下来的原始货品名作为摘要。
- 如果原文中出现 `备注:` 或 `备注：`，会截取其后的内容。
- 最终客服备注为货品摘要加备注文字。
- 客户档案不覆盖客服备注。

## 21. 前端人工修改规则

前端允许人工修改：

- 客户账号
- 业务员
- 快递公司
- 结算方式
- 收款账户
- 客服备注
- 货品名称
- 数量
- 单价

联动规则：

- 修改结算方式为 `欠款计应收` 时，自动清空收款账户。
- 选择收款账户时，自动改为 `银行收款`。
- 清空收款账户时，自动改为 `欠款计应收`。
- 修改货品名称时，自动同步库存里的条码、规格编号和规格。
- 修改货品名称、数量、单价时，自动重新计算合计。
- 修改货品名称后，会触发货品学习。
- 修改客户账号后，可能触发客户学习。

## 24. 当前已发现的未完全生效规则

以下规则在项目中有文件、说明或部分代码，但当前主流程没有完全接上：

- `customer_history.json` 中的老客户购买习惯尚未完整参与主流程匹配。
- 前端有快递学习触发逻辑，但后端启动时没有合并 `express` 学习规则。
- `配置规则.json` 里的快递规则大段配置，与当前实际使用的 `快递表格.xls` 存在重叠。
- 库存表中的成本价、当前成本价没有被用于自动填充销售单价。
- 组合装明细已加入货品索引，但组合装拆分逻辑目前没有明显完整展开。

## 25. 当前核心要求总结

当前项目最重要的业务要求如下：

- 订单原文可以混乱，但最终字段必须结构化。
- 业务员必须优先按括号代码锁定。
- 客户账号必须优先按 `系统录` 相关文字锁定。
- 付款方式和收款账户必须由本地规则判断，不能让 AI 随便猜。
- 货品必须回到库存表中的标准名称，并补全条码、规格、货品编号。
- 快递必须按手动指定、省份、品类、重量依次判断。
- 客服备注必须包含货品摘要，方便仓库核对。
- 导出 Excel 必须分为订单表和货品表，并通过导入编号关联。
- 人工修正后系统应尽量学习，减少下次重复修改。
