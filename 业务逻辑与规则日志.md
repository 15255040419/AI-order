# 智能订单助手 - 业务逻辑与规则日志

本文档用于记录系统的核心业务逻辑、自动识别准则及算价公式，确保后续迭代中逻辑的一致性。

## 1. 核心算价公式
*   **自动算价**：`应收合计 = (单价 * 数量) 之和 + 应收邮资`。
*   **触发机制**：修改“单价”、“数量”或“应收邮资”时，系统会自动实时重算“应收合计”。
*   **手动优先**：如果用户手动强行修改了“应收合计”，系统将保留用户的修改值，不再受公式强制联动，以应对抹零等特殊场景。

## 2. 货品与元数据同步
*   **同步准则**：当在“货品名称”下拉框选择或识别出货品后，系统必须自动同步带出该货品在《总库存.xlsx》中对应的：**货品编号、条码、规格**。
*   **清洁搜索**：系统会自动剥离货品名中的特殊干扰字符（如星号 `*`、空格、括号）进行比对，提高匹配率（如 `57*50*10卷纸` 自动匹配 `57*50纸`）。

## 3. 支付状态与账户逻辑（由系统联动强制执行）
### 3.1 核心设计逻辑
为了防止财务对账时出现“单据标记为未付却填了收款账户”或“给了钱却没选账户”的数据矛盾，系统实现了 **“状态自对齐”** 逻辑。

### 3.2 联动触发机制
*   **【未付】->【收款账户】联动**：
    *   **触发场景**：用户将结算方式选为“**欠款计应收**”。
    *   **联动结果**：系统自动调用 `updateOrder` 策略，强行将 `receipt`（收款账户）字段重置为空白。
    *   **视觉逻辑**：界面实时监测到状态为“欠款”，通过 CSS 条件渲染将对应组件背景设为 **#FEFCE8 (Yellow-100)**，起到强烈的视觉警示作用。
*   **【收款账户】->【结算方式】联动**：
    *   **场景 A（选定账户）**：用户从下拉框选择了具体的收款账户名。
    *   **逻辑结果**：系统推断“既有账户必有进账”，自动将 `payMethod` 切换为“**银行收款**”，并恢复白色背景。
    *   **场景 B（清空账户）**：用户手动将账户点选回“**选择账户**”（即 Value 为空）。
    *   **逻辑结果**：系统推断“既无账户必无进账”，自动将状态打回“**欠款计应收**”，并重新触发黄色警告色。

### 3.3 数据补位逻辑
*   如果 AI 在初步解析时未识别到账户，但识别到了“系统录 XXX”，则该名字会被优先锁定在“客户账号”字段。只有在支付状态确实为“已付”时，才会尝试查询该名字是否属于收款账户。

## 4. 身份与代码识别
*   **业务员代码**：严格识别包裹在括号、全角括号内的单字母或编号（如 `(W)`, `（L）`）。该识别结果直接对应系统业务员。
*   **客户账号**：优先从“系统录”关键字后方提取非空字符作为客户账号（即订单主体名）。
*   **元数据保留**：对于 `[1783]`、`(回家吃饭)` 等位于姓名、地址或电话后的方括号/圆括号内容，视为**原始文字备注**，必须原样保留在对应字段内，严禁将其当做客户编号或业务代码拆分。

## 5. 导出与存储规则
*   **导入编号**：导出 Excel 时，每行数据的“导入编号”固定使用对应订单的**手机号**，以便于后续在导入 ERP 或物流系统时进行唯一性校验。
*   **动态文件名**：生成 Excel 的文件名末尾必须包含单数统计（如：`订单分表_日期_共X单.xlsx`）。

## 6. UI 交互准则
*   **视觉一致性**：所有采集框（Input）和选择框（Select）统一采用白色背景、浅灰色边框及微阴影样式，保持视觉整齐。
*   **多任务解析**：支持不间断粘贴订单，解析过程在后台异步进行，不阻塞用户继续录入下一单。
*   **模糊选中**：前端下拉框支持模糊容错。即便解析出的结果带了前缀/后缀杂质，下拉框应自动选中最接近的选项。
