# 今日规则修改记录

本文档记录本次针对“AI 反复识别错货品、快递、业务员、客户账号”的修复思路和实际改动。

## 1. 原来的问题

原系统存在一个核心问题：让 AI 或模糊匹配参与了太多最终业务判断。

主要风险：

- AI 会把订单里的货品名改写成它认为的标准名称。
- 货品匹配里有别名、包含匹配、模糊匹配，容易把相似货品认错。
- 普通货品如果差一个字、少一个规格，都可能是另一个 SKU。
- 同一个货品名称可能对应不同规格，不能只靠货品名称判断。
- 未确认货品时继续自动判定快递，可能导致快递也跟着错。

所以本次修改的原则是：

```text
AI 只切订单原文，不决定最终货品。
货品只按你的表格规则精确查。
查不到就待确认，不自动猜。
```

## 2. 本次货品规则修复

### 收银机和一体称

收银机和一体称只查：

```text
order-helper-backend/data/总库存-组合装明细.xlsx
```

使用字段：

```text
货品名称
```

规则：

- 订单里的机器货品文字必须精确等于组合装明细表中的 `货品名称`。
- 命中后自动匹配。
- 不做模糊匹配。
- 不做包含匹配。
- 不使用别名。

### 其他普通货品

普通货品只查：

```text
order-helper-backend/data/总库存.xlsx
```

使用字段：

```text
规格编号
```

规则：

- 订单里的普通货品必须写到总库存表里的 `规格编号`。
- 系统只用 `规格编号` 精确匹配。
- 不用总库存表的 `货品名称` 自动匹配普通货品。
- 不用货品编号、条码、别名、模糊匹配自动推断普通货品。

示例：

```text
XP-58TS        -> 不自动匹配
XP-58TS(USB)  -> 匹配到 XP-58TS
```

## 3. 本次 AI 规则修复

AI 现在只负责从订单里截取货品原文，不负责改写标准货品名。

原来容易出现：

```json
{
  "name": "标准货品名"
}
```

现在要求：

```json
{
  "raw_name": "订单原文里的货品文字"
}
```

规则：

- AI 不允许补字。
- AI 不允许改字。
- AI 不允许把货品改写成候选货品。
- 最终货品由本地表格精确匹配决定。

## 4. 未匹配货品处理

如果货品没有按上述规则精确命中：

- 后端标记 `needsReview = true`。
- 前端显示 `[待确认]`。
- 货品行显示黄色提示。
- 用户需要手动从全库搜索里选择正确货品。

系统不会再自信地猜一个相似货品。

## 5. 快递规则修复

快递必须建立在“已确认货品”基础上。

本次修改后：

- 只要订单里存在未确认货品，快递公司直接显示 `待确认`。
- 快递原因显示：`存在未精确匹配货品，确认货品后再判定快递`。
- 未确认货品不会参与重量、品类、快递判断。

这样可以避免：

```text
货品识别错 -> 品类错 -> 重量错 -> 快递错
```

### 5.1 快递表列名修复

本次又修复了一个快递表读取问题。

原问题：

- `快递表格.xls` 里打印机重量段对应列是：
  - `打印机`
  - `Unnamed: 2`
  - `Unnamed: 3`
- 但代码判断时读取的是：
  - `打印机_小`
  - `打印机_中`
  - `打印机_大`
- 导致 2-5kg、5kg 以上没有读到正确列，最后兜底成 `中通（渠道）`。

修复后映射：

| Excel 原列 | 系统规则字段 |
| --- | --- |
| 打印机 | 打印机_小 |
| Unnamed: 2 | 打印机_中 |
| Unnamed: 3 | 打印机_大 |
| 收银机 | 收银机 |
| 一体称 | 一体称 |

示例：

```text
山东 XP-80U(USB+网口)*3
总重约 2.98kg
按山东 2-5kg 列 -> 圆通（渠道）
```

注意：

```text
江苏 JY-335C*2
总重约 7.69kg
按江苏快递表 5kg以上列 -> 中通（渠道）
```

如果这类订单应走其他快递，需要修改 `快递表格.xls` 中江苏 5kg 以上对应单元格。

## 5.2 价格识别修复

原问题：

- 货品单价主要依赖 AI 返回。
- AI 没返回单价时，前端货品单价显示为 `0`。

修复后：

系统会从订单原文中识别确定性价格公式。

支持示例：

```text
90*2=180元
95*3=285元
3*95=285元
90+210=300
```

规则：

- 如果公式里的数量等于当前货品数量，并且单价 × 数量等于合计金额，就回填单价。
- 例如 `XP-80U(USB+网口)*3 95*3=285元`，单价自动回填 `95`。
- 例如 `JY-335C黑*1+S-H80（USB+网口+串口）*1 90+210=300`，会按货品顺序回填 `90` 和 `210`。
- 这属于本地规则，不依赖 AI。

## 5.3 规格编号括号归一化修复

原问题：

- 普通货品只查 `总库存.xlsx` 的 `规格编号` 是正确方向。
- 但订单里可能写中文括号，例如 `S-H80（USB+网口+串口）`。
- 表格里的规格编号可能是英文左括号、中文右括号等混合形式，例如 `S-H80(USB+网口+串口）`。
- 旧扫描逻辑直接用原字符串包含判断，括号形态不一致时会漏识别。

修复后：

- 规格编号匹配前会做轻量归一化。
- 中文括号和英文括号视为同一种括号。
- 空格差异不影响匹配。
- 仍然是精确匹配，不做模糊、不做别名、不做包含猜测。

示例：

```text
订单：S-H80（USB+网口+串口）*1
表格：S-H80(USB+网口+串口）
结果：匹配到 佳博S-H80
```

## 5.4 客服备注多货品连接符修复

多货品订单的客服备注中，货品摘要之间使用 `+` 连接。

示例：

```text
JY-335C黑*1+S-H80（USB+网口+串口）*1 咸阳市泾阳县崇文镇文定天下店
```

不再使用空格连接多个货品摘要。

## 5.5 补运费识别修复

原问题：

- `98*2+95+10=301元` 这类公式能表示“两个货品金额 + 补运费”。
- 之前系统只把 `301元` 识别为应收合计，没有把最后的 `10` 写入应收邮资。

修复后：

```text
XP-80U(USB+网口)*2+JY-335C黑*1+补运费 98*2+95+10=301元
```

会识别为：

```text
XP-80U 单价 98，数量 2
JY-335C 单价 95，数量 1
应收邮资 10
应收合计 301
```

同时修复了本地预扫描货品顺序问题：多个货品会按订单原文出现顺序排序，确保价格公式按顺序分配。

后续增强：

```text
A*2+补运费+B*1 98*2+10+95=301元
```

如果 `补运费` 写在第二个位置，则第二个价格项 `10` 会写入应收邮资；货品价格继续按剩余位置对应。

数量识别增强：

```text
XP-80U(USB+网口)**2
```

现在会识别为数量 `2`。系统支持 `*2`、`**2`、`x2`、`×2` 等数量写法。

## 9. 后续如何新增收款账户

如果以后新增一个已付账户，不需要改 AI。

只需要改：

```text
order-helper-backend/data/配置规则.json
```

找到：

```json
"收款账户映射": {
}
```

新增一行关键词映射即可。

示例：

```json
"已付新账户": "新账户名称"
```

规则说明：

- 左边是订单原文里会出现的关键词。
- 右边是最终写入系统的收款账户名称。
- 系统会按关键词长度倒序匹配，长关键词优先。
- 只要命中收款账户关键词，结算方式会自动变成 `银行收款`。

新增后重启后端服务即可生效。

## 10. 后续如何新增快递公司

新增快递通常分两部分。

### 10.1 新增前端可选项

修改：

```text
order-helper-backend/data/配置规则.json
```

找到：

```json
"快递选项": [
]
```

把新快递名称加进去。

示例：

```json
"申通（渠道）"
```

这样前端下拉框里会出现新快递。

### 10.2 新增自动判定规则

如果只是让用户手动选择，加 `快递选项` 就够了。

如果希望系统自动判定，需要改快递规则来源：

```text
order-helper-backend/data/快递表格.xls
```

当前快递自动判定主要来自这个表。

需要按省份、重量段、收银机、一体称等列更新对应快递名称。

新增或修改后，重启后端服务即可重新读取。

### 10.3 新增“原文指定快递”关键词

如果以后订单原文里会写：

```text
发申通
走申通
指定申通
```

还需要修改：

```text
order-helper-backend/core/express.py
```

在 `manual_keywords` 里新增：

```python
"申通": "申通（渠道）"
```

这样原文指定申通时，系统会优先使用申通。

## 11. 后续如何新增货品规则

目前不建议新增“货品别名规则”。

你的货品规则已经明确：

- 收银机/一体称：维护 `总库存-组合装明细.xlsx` 的 `货品名称`。
- 普通货品：维护 `总库存.xlsx` 的 `规格编号`。

以后新增货品时：

- 机器类货品加到组合装明细表，并保证 `货品名称` 唯一。
- 普通货品加到总库存表，并保证 `规格编号` 唯一。

订单里只要按这两个字段填写，系统就能稳定识别。

## 12. 后续新增规则的推荐方式

以后新增业务规则时，建议按下面原则处理：

```text
死规则写配置或表格。
AI 不参与最终判断。
查不到就待确认。
人工修正后再决定是否沉淀成明确规则。
```

优先修改位置：

| 规则类型 | 修改位置 |
| --- | --- |
| 新业务员 | `配置规则.json` 的 `业务员映射` |
| 新收款账户 | `配置规则.json` 的 `收款账户映射` |
| 新快递下拉选项 | `配置规则.json` 的 `快递选项` |
| 快递省份/重量规则 | `快递表格.xls` |
| 原文指定快递关键词 | `core/express.py` 的 `manual_keywords` |
| 新普通货品 | `总库存.xlsx` 的 `规格编号` |
| 新收银机/一体称 | `总库存-组合装明细.xlsx` 的 `货品名称` |
