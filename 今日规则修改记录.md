# 今日规则修改记录

本文档记录本次针对“AI 反复识别错货品、快递、业务员、客户账号”的修复思路和实际改动。

## 1. 原来的问题

原系统存在一个核心问题：让 AI 或模糊匹配参与了太多最终业务判断。

主要风险：

- AI 会把订单里的货品名改写成它认为的标准名称。
- 货品匹配里有别名、包含匹配、模糊匹配，容易把相似货品认错。
- 普通货品如果差一个字、少一个规格，都可能是另一个 SKU。
- 同一个货品名称可能对应不同规格，不能只靠货品名称判断。
- 未确认货品时继续自动判定快递，可能导致快递也跟着错。

所以本次修改的原则是：

```text
AI 只切订单原文，不决定最终货品。
货品只按你的表格规则精确查。
查不到就待确认，不自动猜。
```

## 2. 本次货品规则修复

### 收银机和一体称

收银机和一体称只查：

```text
order-helper-backend/data/总库存-组合装明细.xlsx
```

使用字段：

```text
货品名称
```

规则：

- 订单里的机器货品文字必须精确等于组合装明细表中的 `货品名称`。
- 命中后自动匹配。
- 不做模糊匹配。
- 不做包含匹配。
- 不使用别名。

### 其他普通货品

普通货品只查：

```text
order-helper-backend/data/总库存.xlsx
```

使用字段：

```text
规格编号
```

规则：

- 订单里的普通货品必须写到总库存表里的 `规格编号`。
- 系统只用 `规格编号` 精确匹配。
- 不用总库存表的 `货品名称` 自动匹配普通货品。
- 不用货品编号、条码、别名、模糊匹配自动推断普通货品。

示例：

```text
XP-58TS        -> 不自动匹配
XP-58TS(USB)  -> 匹配到 XP-58TS
```

## 3. 本次 AI 规则修复

AI 现在只负责从订单里截取货品原文，不负责改写标准货品名。

原来容易出现：

```json
{
  "name": "标准货品名"
}
```

现在要求：

```json
{
  "raw_name": "订单原文里的货品文字"
}
```

规则：

- AI 不允许补字。
- AI 不允许改字。
- AI 不允许把货品改写成候选货品。
- 最终货品由本地表格精确匹配决定。

## 4. 未匹配货品处理

如果货品没有按上述规则精确命中：

- 后端标记 `needsReview = true`。
- 前端显示 `[待确认]`。
- 货品行显示黄色提示。
- 用户需要手动从全库搜索里选择正确货品。

系统不会再自信地猜一个相似货品。

## 5. 快递规则修复

快递必须建立在“已确认货品”基础上。

本次修改后：

- 只要订单里存在未确认货品，快递公司直接显示 `待确认`。
- 快递原因显示：`存在未精确匹配货品，确认货品后再判定快递`。
- 未确认货品不会参与重量、品类、快递判断。

这样可以避免：

```text
货品识别错 -> 品类错 -> 重量错 -> 快递错
```

### 5.1 快递表列名修复

本次又修复了一个快递表读取问题。

原问题：

- `快递表格.xls` 里打印机重量段对应列是：
  - `打印机`
  - `Unnamed: 2`
  - `Unnamed: 3`
- 但代码判断时读取的是：
  - `打印机_小`
  - `打印机_中`
  - `打印机_大`
- 导致 2-5kg、5kg 以上没有读到正确列，最后兜底成 `中通（渠道）`。

修复后映射：

| Excel 原列 | 系统规则字段 |
| --- | --- |
| 打印机 | 打印机_小 |
| Unnamed: 2 | 打印机_中 |
| Unnamed: 3 | 打印机_大 |
| 收银机 | 收银机 |
| 一体称 | 一体称 |

示例：

```text
山东 XP-80U(USB+网口)*3
总重约 2.98kg
按山东 2-5kg 列 -> 圆通（渠道）
```

注意：

```text
江苏 JY-335C*2
总重约 7.69kg
按江苏快递表 5kg以上列 -> 中通（渠道）
```

如果这类订单应走其他快递，需要修改 `快递表格.xls` 中江苏 5kg 以上对应单元格。

## 5.2 价格识别修复

原问题：

- 货品单价主要依赖 AI 返回。
- AI 没返回单价时，前端货品单价显示为 `0`。

修复后：

系统会从订单原文中识别确定性价格公式。

支持示例：

```text
90*2=180元
95*3=285元
3*95=285元
90+210=300
```

规则：

- 如果公式里的数量等于当前货品数量，并且单价 × 数量等于合计金额，就回填单价。
- 例如 `XP-80U(USB+网口)*3 95*3=285元`，单价自动回填 `95`。
- 例如 `JY-335C黑*1+S-H80（USB+网口+串口）*1 90+210=300`，会按货品顺序回填 `90` 和 `210`。
- 这属于本地规则，不依赖 AI。

## 5.3 规格编号括号归一化修复

原问题：

- 普通货品只查 `总库存.xlsx` 的 `规格编号` 是正确方向。
- 但订单里可能写中文括号，例如 `S-H80（USB+网口+串口）`。
- 表格里的规格编号可能是英文左括号、中文右括号等混合形式，例如 `S-H80(USB+网口+串口）`。
- 旧扫描逻辑直接用原字符串包含判断，括号形态不一致时会漏识别。

修复后：

- 规格编号匹配前会做轻量归一化。
- 中文括号和英文括号视为同一种括号。
- 空格差异不影响匹配。
- 仍然是精确匹配，不做模糊、不做别名、不做包含猜测。

示例：

```text
订单：S-H80（USB+网口+串口）*1
表格：S-H80(USB+网口+串口）
结果：匹配到 佳博S-H80
```

## 5.4 客服备注多货品连接符修复

多货品订单的客服备注中，货品摘要之间使用 `+` 连接。

示例：

```text
JY-335C黑*1+S-H80（USB+网口+串口）*1 咸阳市泾阳县崇文镇文定天下店
```

不再使用空格连接多个货品摘要。

## 5.5 补运费识别修复

原问题：

- `98*2+95+10=301元` 这类公式能表示“两个货品金额 + 补运费”。
- 之前系统只把 `301元` 识别为应收合计，没有把最后的 `10` 写入应收邮资。

修复后：

```text
XP-80U(USB+网口)*2+JY-335C黑*1+补运费 98*2+95+10=301元
```

会识别为：

```text
XP-80U 单价 98，数量 2
JY-335C 单价 95，数量 1
应收邮资 10
应收合计 301
```

同时修复了本地预扫描货品顺序问题：多个货品会按订单原文出现顺序排序，确保价格公式按顺序分配。

后续增强：

```text
A*2+补运费+B*1 98*2+10+95=301元
```

如果 `补运费` 写在第二个位置，则第二个价格项 `10` 会写入应收邮资；货品价格继续按剩余位置对应。

数量识别增强：

```text
XP-80U(USB+网口)**2
```

现在会识别为数量 `2`。系统支持 `*2`、`**2`、`x2`、`×2` 等数量写法。

## 13. 流式解析与前端交互体验优化 (最新修改)

本次未提交的修改主要集中在**前端订单界面 (`order-assistant-full.html`)** 和 **后端解析服务 (`order-helper-backend/app.py`)**，主要实现了**流式解析进度反馈**、**细粒度局部渲染（彻底解决滚动跳动和失焦问题）** 以及**视觉交互体验的全面升级**。

### 13.1 后端 (`app.py`) 修改内容与逻辑

*   **新增流式解析接口 (`/api/parse-stream`)**
    *   **做了什么**: 引入了 `flask` 的 `Response` 和 `stream_with_context`，结合 `threading` 和 `queue` 将原本同步的订单解析主逻辑提取到 `process_order_text` 并放入后台线程运行。执行过程中，使用内部函数 `_progress` 将各解析阶段（如“本地预解析”、“调用 AI 解析”、“精确匹配”等）作为 `progress` 事件推送到队列，最终通过 Server-Sent Events (SSE) 格式实时返回给前端。
    *   **逻辑/目的**: 原本的 `/api/parse` 在处理复杂订单时可能耗时较长，前端只能处于盲目的等待状态。引入流式响应后，前端可以实时感知后端的处理进度，极大缓解了用户的等待焦虑。
*   **增加解析追踪记录 (Debug Trace)**
    *   **做了什么**: 在解析引擎的每一步中，搜集关键执行数据（如耗时、匹配规则、上下文信息、最终触发的快递判定逻辑等），封装进 `result["debugTrace"]` 中一并返回。
    *   **逻辑/目的**: 增强系统的透明度与可观测性。遇到识别不准的订单时，可以通过追踪数据明确问题是出在本地预扫描、AI 提取阶段，还是后续的业务员匹配与快递判定规则中，极大方便了后续的系统调优排错。

### 13.2 前端 (`order-assistant-full.html`) 修改内容与逻辑

*   **重构渲染逻辑，实现细粒度局部更新 (重点修复性能与交互)**
    *   **做了什么**: 彻底抛弃了此前只要有状态更新（如修改任意字段、打开下拉框）就调用全局 `render()` 重建整个 DOM 树的做法。新增了基于 `requestAnimationFrame` 调度的局部更新方法（如 `replaceOrderCard`、`scheduleOrderCardRefresh`）。现在如果修改了某个订单的信息，只会精准替换页面中对应的这单个订单卡片。对于必须全量渲染的场景（如清空、删除），引入了 `renderKeepScroll` 来记录并恢复滚动快照。
    *   **逻辑/目的**: 解决了此前痛点——**操作订单时页面总是意外重置滚动条位置或丢失输入焦点**。局部渲染让操作变得如丝般顺滑，不再卡顿跳动。
*   **引入流式加载界面与 Fallback (降级) 机制**
    *   **做了什么**: `handleSubmitText` 提交文本后不再等待接口整体返回，而是立即在列表末尾插入占位卡片，并调用 `parseOrderWithStream` 发起流式请求。前端通过 `TextDecoder` 读取 SSE 流数据，监听 `progress` 事件并实时更新卡片上的文案。为了保证容错，如果流式连接建立失败，会自动回退 (`Fallback`) 到普通同步解析模式。
    *   **逻辑/目的**: 配合后端提供“所见即所得”的进度反馈。使用流式加载可以避免批量上传时页面卡死假死的现象。
*   **UI 视觉与交互体验全面升级 (Premium Design)**
    *   **做了什么**:
        *   **重构加载动画**: 废弃了简单的 Loading 圈，改用充满极客感的代码打字机动画（`.cc-type-line`）和进度扫描线（`.cc-scan-row`）。
        *   **动态环境光**: 为页面背景添加了跟随鼠标移动的光晕特效 (`--mouse-x/y` 配合 `radial-gradient` 和 `pointermove` 事件)。
        *   **入场动画与质感**: 订单卡片新增顺滑的滑入动画 (`card-type-in`)；各类按钮、芯片 (`cc-chip`) 使用了 `color-mix` 颜色混合方案配合内阴影，使质感更加通透高级。
        *   **异常订单强提醒**: 为重复订单（相同手机号）添加了专属的 `.duplicate-highlight` 样式——红色的呼吸灯边框和醒目的“重复订单”红色徽标。
    *   **逻辑/目的**: 改善界面的专业感与现代感，摆脱简陋的“后台工具”气息；同时利用设计手段（如动画、高亮警告）更好地引导用户的视线，降低重复发货等业务操作失误。

## 14. 修复本地预扫与 AI 解析冲突及前端背景特效增强 (最新修改)

本次修改解决了**“部分由于错别字或复杂名称导致的货品，在存在其他标准货品时被系统意外丢弃”**的严重 Bug，并为前端页面增加了极具科技感的物理交互特效。

### 14.1 核心 Bug 修复 (`app.py` & `core/ai_parser.py`)

*   **解除“找到一个就抛弃 AI”的短路逻辑**
    *   **问题所在**: 旧代码中存在 `ai_products = local_info["products"] or ai_res.get('products', [])`。这意味着只要本地根据规格编号抓到了一件货（例如 `XP-80TS`），系统就会认为货找齐了，直接把 AI 辛辛苦苦拆出来的其他货品（如错字版 `K3白单`）全部丢进垃圾桶。
    *   **做了什么**: 重构了货品合并逻辑。现在系统会优先拿出 AI 完整拆分出来的所有商品数组，然后再遍历本地提取到的 `local_prods` 进行查漏补缺（按字符串包含关系去重）。这保证了 AI 识别出的任何疑似货品都会被保留并交给后端的 `find_strict_match` 审查，如果查不到再标记为 `[待确认]`，而绝不会直接消失。
*   **解除“对 AI 隐瞒底库”的 Bug**
    *   **问题所在**: 旧代码在本地抓到货品后，直接不再给 AI 提供 `product_ref`（底层候选商品库）。导致 AI 在面对复杂订单时“两眼一抹黑”，极大降低了其切分陌生货品（耗材、纸张等）的准确率。
    *   **做了什么**: 修改 `app.py` 让 `get_candidate_products` 永远执行，不仅归一化处理（去除空格括号）来寻找疑似商品，还在最后暴力截取了前 2000 个商品库名单作为 AI 的兜底参考。同时在 `ai_parser.py` 中，明确排除了将 `local_info` 中的 `products` 传给 AI（防止 AI 偷懒只返回本地已知的这一个）。

### 14.2 前端科技感特效增强 (`order-assistant-full.html`)

*   **新增星网粒子交互背景**
    *   **做了什么**: 在页面最底层（`body` 之上，`.cc-shell` 之下，并将 `.cc-shell` 背景设为透明）注入了一张全屏 Canvas 画布，通过纯 Vanilla JS 编写了 `ParticleSystem` 粒子系统。
    *   **效果描述**:
        *   支持根据系统的日/夜间模式自动切换主色调（继承 `--cc-accent`）。
        *   粒子根据距离互相产生半透明连线。
        *   **物理发散交互**: 增加对鼠标的排斥力与动能继承，当用户在背景上快速滑动鼠标时，粒子会被“物理推开”并继承滑动速度，营造出强烈的操作反馈和极客科技感。
    *   **优化调整**: 根据实际视觉效果，将粒子上限翻倍至 250 个，连线距离提升至 180px，并加深了透明度，使星网更加绵密清晰。
