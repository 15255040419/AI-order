# AI 下单助手

这是一个线下订单录入辅助系统。用户粘贴订单原文后，系统会拆出收货信息、货品、价格、付款方式、客户账号、业务员、快递等字段，并生成可导出的订单表和货品表。

## 核心原则

本项目当前采用：

```text
AI 只负责兜底切割订单原文。
最终业务字段必须由本地规则、Excel 表格、JSON 配置决定。
本地能确定的订单直接跳过 AI。
查不到就待确认，不自动乱猜。
```

禁止恢复以下做法：

- 让 AI 决定最终标准货品。
- 让 AI 决定最终快递。
- 让 AI 决定最终收款账户。
- 让 AI 决定最终业务员。
- 用模糊匹配、包含匹配、别名推断自动选择货品。
- 货品未确认时仍自动判定快递。

## 启动和数据缓存

进入后端目录：

```bash
cd order-helper-backend
```

安装依赖：

```bash
pip install -r requirements.txt
```

启动服务：

```bash
python app.py
```

访问：

```text
http://127.0.0.1:5005
```

后台启动时会自动加载 `order-helper-backend/data/` 下的 Excel 和 JSON 文件，并按源文件修改时间刷新 `.cache/*.pkl` 缓存。

运行中如果你更新了表格，后端现在也会在下一次解析订单前自动检测源文件修改时间并刷新内存数据。也可以手动调用：

```text
POST http://127.0.0.1:5005/api/reload-data
```

注意：如果 Excel 文件正被表格软件锁住，刷新可能失败；最稳妥的做法是保存并关闭表格后再解析或刷新。

## 订单处理流程

1. 用户输入订单原文。
2. 后端先本地识别付款方式、收款账户、手机号、客户账号、业务员代码、货品和数量。
3. 如果手机号、地址、客户账号、货品都能本地确定，并且货品精确命中表格，直接跳过 AI。
4. 如果核心字段不完整，才调用 AI 做兜底切割。
5. 所有货品再次按 Excel 精确匹配，补全规格编号、货品编号、条码、规格、重量等信息。
6. 本地规则生成客户账号、业务员、付款方式、收款账户、客服备注、总金额和快递。
7. 前端展示订单卡片，用户可人工修改。
8. 导出 Excel，生成订单表和货品表。

## 字段分工

AI 只允许负责：

- 收货人初步切分。
- 手机号初步切分。
- 地址初步切分。
- 省份初步切分。
- 货品原文片段切分。
- 数量初步切分。
- 备注原文切分。

本地规则必须负责：

- 客户账号。
- 业务员。
- 付款方式。
- 收款账户。
- 标准货品。
- 规格编号。
- 货品编号。
- 条码。
- 规格。
- 价格。
- 应收邮资。
- 快递公司。
- 客服备注。
- 导出字段。

## 货品匹配规则

货品最终匹配必须本地查表，不能让 AI 猜。

收银机、一体称、组合装只查：

```text
order-helper-backend/data/总库存-组合装明细.xlsx
```

唯一匹配字段：

```text
货品名称
```

普通货品只查：

```text
order-helper-backend/data/总库存.xlsx
```

唯一匹配字段：

```text
规格编号
```

规则：

- 必须精确匹配。
- 不做模糊匹配。
- 不做包含匹配。
- 不用别名自动推断。
- 不用普通货品的 `货品名称` 自动匹配。
- 不用货品编号、条码自动推断。

允许的轻量归一化：

- 中文括号和英文括号视为同一种括号。
- 空格差异忽略。

这仍然是精确匹配，不是模糊匹配。

示例：

```text
57*50纸       -> 匹配总库存.xlsx 的规格编号 57*50纸
JY-335C黑     -> 匹配总库存.xlsx 的规格编号 JY-335C黑
XP-58TS       -> 不自动匹配，除非表格规格编号就是 XP-58TS
XP-58TS(USB)  -> 匹配表格中对应规格编号
```

如果货品无法精确匹配：

- 后端设置 `needsReview = true`。
- 前端显示 `[待确认]`。
- 快递显示 `待确认`。
- 用户需要手动从全库搜索选择。

## 客户账号规则

客户账号优先来自订单原文或客户档案表，不依赖 AI 判断。

规则：

- 如果原文出现 `系统录`、`系统录入`、`系统录制`、`系统记录`，取其后内容作为客户账号。
- 如果写法是 `系统录 河南慧通电子商务（W）`，客户账号为 `河南慧通电子商务`，业务员代码为 `W`。
- 如果 `系统录` 后没有括号业务员代码，使用客户档案业务员；仍没有则使用默认业务员。
- 如果手机号命中客户档案，可回填客户账号。
- 如果原文直接出现客户档案里的客户账号或客户名称，可辅助回填客户账号。
- AI 返回的 `customer_account` 只能兜底，不能覆盖本地命中结果。

## 业务员规则

业务员不由 AI 决定。

优先级：

1. 原文括号代码，例如 `(L)`、`（D）`、`(TX)`、`（W）`。
2. 客户档案中的业务员。
3. 默认业务员。

配置位置：

```text
order-helper-backend/data/配置规则.json -> 业务员映射
```

当前常用代码：

| 代码 | 业务员 |
| --- | --- |
| TX | 仝心科技 |
| L | 陆香 |
| D | 王德龙 |
| C | 王德成 |
| W | 汪朋松 |
| T | 胡正婷 |
| H | 韩伟 |
| F | 方同辉 |

## 付款方式和收款账户

付款方式和收款账户由本地规则判断。

付款方式：

- 出现 `已付`、`转账`、`已转` -> `银行收款`。
- 出现 `未付`、`欠款` -> `欠款计应收`。
- 命中具体收款账户时，强制为 `银行收款`。
- 默认 -> `欠款计应收`。

配置位置：

```text
order-helper-backend/data/配置规则.json -> 收款账户映射
```

匹配方式：

- 按关键词长度倒序匹配。
- 长关键词优先，避免短关键词误伤长关键词。
- AI 不参与收款账户判断。

示例：

```text
已付农商龙   -> 农商（龙）
已付安徽仝心 -> 仝心农商（公账）
```

## 价格规则

价格优先从订单原文本地解析。

支持：

```text
90*2=180元
95*3=285元
3*95=285元
90+210=300
98*2+95+10=301元
单品 880元
```

规则：

- 乘法公式用于识别单个货品单价。
- 加法公式按货品出现顺序对齐。
- `补运费`、`邮费`、`快递费` 进入应收邮资，不作为货品。
- 单品直写金额可作为单价或总价兜底。
- 0 元赠品必须保留为 0，不能被兜底逻辑覆盖。

## 快递规则

快递不由 AI 决定。

优先级：

1. 原文手动指定快递。
2. 物流单号前缀。
3. 快递表格中的省份规则。
4. 货品品类。
5. 重量分段。
6. 默认快递。

快递表：

```text
order-helper-backend/data/快递表格.xls
```

关键规则：

- 只要存在未确认货品，快递必须是 `待确认`。
- 未确认货品不能参与重量和品类判断。
- 手动指定快递优先，例如 `发顺丰`、`走德邦`、`指定圆通`。
- `JT` 单号前缀可识别为极兔。

## 客服备注和物流单号

客服备注由本地合成，不直接采用 AI 自由生成。

格式：

```text
货品原始摘要 + 原文备注内容
```

多个货品之间用 `+` 连接：

```text
JY-335C黑*1+S-H80（USB+网口+串口）*1 备注内容
```

如果原文出现 `备注:` 或 `备注：`：

- 备注之后的内容只进入客服备注。
- 不应再被识别为正式货品。

物流单号规则：

```text
单号录：JT2184157335752 -> trackingNumber = JT2184157335752
```

## 多单性能规则

慢的根因不是表格大，而是每个订单都同步等一次 AI 接口。

当前优化：

- 本地规则能确定核心字段时，订单直接跳过 AI。
- 货品和客户匹配走规范化索引，不再每次全表扫描。
- 多单一起发时，规则完整的订单应接近本地毫秒级处理。
- AI 只处理地址或货品字段不完整的疑难订单。

本地快速路径必须同时满足：

- 有手机号。
- 能从手机号前后切出收货人和地址。
- 能识别客户账号。
- 至少有一个货品。
- 所有货品都精确命中 Excel。

## 修改前检查清单

后续修改前必须确认：

- 这个字段是不是死规则？
- 这个字段是不是已经在 Excel 或 JSON 里？
- 能不能通过精确查表解决？
- 查不到时是不是应该待确认？
- 改动会不会让 AI 重新参与最终判断？
- 改动会不会让普通货品重新走模糊、包含或别名匹配？

如果改动会让 AI 猜最终业务字段，应停止这种修改。
